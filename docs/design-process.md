# Design Process

## Цепочка

```text
Approved brief
→ real references
→ ChatGPT UI reference
→ Google Stitch prototype and iterations
→ optional Figma export and cleanup
→ approved desktop + mobile design
→ ecosystem audit
→ Codex implementation plan
→ implementation and browser comparison
```

## Источник истины по этапам

- Product scope и контент: `docs/brief.md`.
- Принятые решения: `docs/decisions.md`.
- Визуальный результат после Phase 5: утверждённые Figma frames.
- Если экспорт в Figma бесполезен: утверждённые Stitch screens плюс зафиксированные design tokens становятся визуальным источником истины; решение записывается в `decisions.md`.
- Код Stitch не является источником архитектуры без отдельного аудита.

## ChatGPT reference

- Генерировать реалистичный UI, а не рекламный арт.
- Сначала выбрать одно направление из максимум 2–3 вариантов.
- Подготовить desktop overview, ключевые секции и mobile-направление.
- Не требовать пиксельной точности от сгенерированного изображения.

## Google Stitch

- Передать brief, выбранные референсы, структуру, visual rules и anti-references.
- Использовать Stitch для композиции, вариантов и прототипа, не считать первый результат финальным.
- Сделать только необходимые итерации до цельного desktop/mobile решения.
- Не добавлять новые секции и функции без изменения brief.
- Phase 4 artifact: `https://stitch.withgoogle.com/projects/1893122908879395195`.
- На входе в Figma проверить одинаковый состав пяти услуг и options формы на desktop/mobile.

## Figma

**Phase 5 result:** экспорт принят. В Figma находятся два отдельных полных frame — desktop и mobile. Они являются визуальным источником истины для Phase 6–10.

Экспортировать, только если результат остаётся редактируемым и экономит время. После экспорта проверить:

- структуру frames и Auto Layout;
- сетку и контейнер;
- повторяемые компоненты;
- цветовые и текстовые стили;
- spacing, состояния CTA и формы;
- desktop и mobile layouts.

Не тратить время на невидимую «идеальность» слоёв, если она не помогает реализации.

Для текущего portfolio demo ручная перестройка экспортированных frames, компонентов и Auto Layout не требуется. Точные размеры, tokens и повторяемые компоненты будут зафиксированы в Phase 6–7 по утверждённому визуальному результату.

## Критерии утверждения desktop

- За первый экран понятны услуга, география, отличие и главное действие.
- Секции образуют один conversion flow.
- Есть понятная сетка, иерархия и единый визуальный язык.
- Контент реалистичен; нет lorem ipsum и фиктивных доказательств.
- Все элементы выполнимы без тяжёлой или хрупкой реализации.

## Критерии утверждения mobile

- Это отдельная продуманная композиция, а не уменьшенный desktop.
- CTA доступен и не перекрывает контент.
- Навигация, карточки, FAQ и форма удобны для касания и клавиатуры.
- Текст читается без масштабирования; горизонтального скролла нет.
- Порядок секций сохраняет главный пользовательский сценарий.

## Правила использования кода Stitch

Код можно взять за основу только если короткий аудит подтверждает:

- совместимость с утверждённым стеком;
- ясную компонентную структуру без лишних абстракций;
- отсутствие ненужных зависимостей;
- нормальную семантику и accessibility;
- отсутствие жёсткой привязки к одному размеру экрана;
- заметную экономию времени относительно чистой реализации.

Если хотя бы один ключевой пункт не выполнен, код используется только как reference для структуры и стилей. Не выполнять большой рефакторинг плохого экспорта ради его сохранения.
