# Modern Cleaning Business Website — правила работы

## Источники истины

1. `docs/brief.md` определяет цели, аудиторию, scope и критерии готовности.
2. Утверждённые desktop- и mobile-макеты в Figma определяют внешний вид.
3. `docs/decisions.md` хранит принятые решения.
4. `docs/plan.md` определяет текущую фазу и порядок работ.

При конфликте остановиться и запросить решение. Не додумывать требования.

## Режим работы

- Всегда использовать Ponytail (`full`): минимальный scope, минимум зависимостей и повторного анализа.
- Работать только над одной активной фазой из `docs/plan.md`.
- Не начинать реализацию до утверждения brief, desktop- и mobile-дизайна.
- Не создавать код, зависимости, инфраструктуру и деплой на дизайн-фазах.
- Не расширять архитектуру и scope без явного подтверждения.
- Утверждённое решение записывать в `docs/decisions.md` и не пересматривать без новых фактов.
- Задавать вопросы одним компактным пакетом; не спрашивать то, что уже записано.

## Skills и инструменты

- UI/UX Pro Max использовать только для выбора визуального направления и финального UI-review.
- Ponytail использовать постоянно.
- Любую новую библиотеку, skill, MCP или сервис сначала обосновать: задача, простой вариант без него, цена сложности и риски. Устанавливать только после подтверждения.
- Код Google Stitch считать референсом, пока отдельный аудит не подтвердит его пригодность.

## Проверки

- Во время реализации запускать только узкие проверки изменённого сценария.
- Полный responsive, accessibility, performance, SEO и release QA проводить в Phase 11.
- Не повторять полный аудит после мелких изменений.

## Формат отчёта после каждого запуска

1. Что завершено.
2. Какие файлы изменены.
3. Какие решения приняты.
4. Один рекомендуемый следующий шаг.
